# 核心任務（CORE-*）

- Task 版本: v1.0
- 狀態: 草稿（尚未開工）
- 說明: 核心任務是所有模組任務（`01`~`09`）的地基，**必須依序完成**（CORE-1 → 2 → 3/4 → 5/6 → 7），
  同一時間只允許一個 subagent 動 CORE 任務，避免地基層互相踩踏。核心任務完成前，模組任務只能做「不依賴
  地基」的準備工作（讀規格、寫 Resource 定義草稿），不能寫會跑的整合程式碼。

每個任務欄位定義：**目標**（為什麼做）／**產出**（具體檔案/可驗證成果）／**規格來源**（依據哪份 spec）／
**前置依賴**／**驗收標準**／**擁有檔案**（其他任務不可直接改動，見 `11_並行協作規則.md`）。

---

## CORE-1　Godot 專案骨架

- **目標**：建立可開啟、可執行的空白 Godot 4.3+ 專案，設定跟 GDevelop 版一致的視窗/渲染參數，之後所有任務
  都在這個骨架上疊加。
- **產出**：
  - `godot-project/project.godot`（視窗 1280×720、2D 像素渲染設定、輸入對應表 InputMap）
  - `godot-project/scenes/` `scripts/` `resources/content/` `assets/` `tests/` 目錄結構（已預先建好空目錄）
  - `.gitignore`（排除 `.godot/`、`export_presets.cfg` 內的簽章資訊等）
  - 一個空場景 `scenes/title/title.tscn` 可以在編輯器開啟並執行不報錯
- **規格來源**：`CLAUDE.md`「Godot 版本與技術選型」一節（`game.json` L`properties`：windowWidth 1280,
  windowHeight 720, scaleMode nearest, antialiasingMode none, pixelsRounding true）
- **前置依賴**：無
- **驗收標準**：`godot --headless --check-only --path godot-project` 通過；編輯器內執行 title 場景無
  error/warning。
- **擁有檔案**：`godot-project/project.godot`、`godot-project/.gitignore`、頂層目錄結構本身。

---

## CORE-2　CONTENT.json → Godot Resource 轉存

- **目標**：把 `../gd-crystal-tales/projects/crystal-quest/CONTENT.json`（party/equipment/skills/enemies/
  encounters/shops/chests/pacing）轉成 Godot 端可用的資料類別與 `.tres`/執行期讀取路徑，作為**唯一資料源**，
  之後改數值只改 CONTENT.json，不手動同步 Godot 端資料。
- **產出**：
  - `godot-project/scripts/content/` 下的 Resource 類別定義：`PartyMemberDef`, `EquipmentDef`, `SkillDef`,
    `ItemDef`, `EnemyDef`, `EncounterDef`, `ShopDef`, `ChestDef`, `DerivedParams`, `PacingParams`
  - 一支轉存腳本（可以是 Python，跟現有 `art_v*.py` 系列一致風格，或 Godot `@tool` script）：讀
    `CONTENT.json` → 輸出 `godot-project/resources/content/*.tres`（或執行期直接 parse JSON，兩種方式
    擇一，取決於 CORE-2 實作時對「build-time 轉存」vs「run-time 讀取」的取捨，定案後回填本節）
  - `ContentDB`（autoload）：提供 `ContentDB.get_enemy(id)`／`ContentDB.get_skill(id)` 等查詢介面，模組任務
    一律透過這層介面拿資料，不直接讀檔案。
- **規格來源**：`../gd-crystal-tales/projects/crystal-quest/CONTENT.json`（現有 schema，561 行，8 個頂層
  分類）
- **前置依賴**：CORE-1
- **驗收標準**：`ContentDB` 載入後，`get_enemy("goblin_boss")` 等既有 id 能拿到跟 CONTENT.json 原始資料
  一致的值（寫一支對照腳本核對數量/關鍵欄位）。
- **擁有檔案**：`godot-project/scripts/content/**`、`godot-project/autoload/content_db.gd`、
  `godot-project/resources/content/**`。

## CORE-3　存檔系統

- **目標**：實作 `SaveManager` autoload，schema 照搬 `specs/SAVE_SCHEMA.md`。
- **產出**：`godot-project/autoload/save_manager.gd`，含 `save_game()`/`load_game()`/`has_save()`/
  `delete_save()`，存檔路徑 `user://cq_save.json`。
- **規格來源**：`specs/SAVE_SCHEMA.md`（全文）
- **前置依賴**：CORE-1、CORE-4（需要 `GameState` 提供要序列化的欄位）——**與 CORE-4 互相依賴，建議同一位
  subagent 或緊接著的兩個任務連續完成，不要拆給不同 agent 平行做**。
- **驗收標準**：存檔→重啟編輯器內跑的 headless 測試場景→讀檔，`GameState` 還原內容與存檔前一致（含
  室內存門口外座標的規則）。
- **擁有檔案**：`godot-project/autoload/save_manager.gd`。

---

## CORE-4　全域狀態 Autoload（GameState）

- **目標**：取代 GDevelop 的全域變數（`g_party`/`g_flags`/`g_eqInv`/`g_itemInv`/`g_gold`/`g_chests`/
  `g_autoBattle` 等），提供**原生型別**（不要維持 GDevelop 版「一切都是 JSON 字串」的怪癖）。
- **產出**：`godot-project/autoload/game_state.gd`，欄位對照 `specs/SAVE_SCHEMA.md` 的全域變數表；並提供
  `invAdd`/`invGet`/`invAll`/`invUse` 等背包操作介面（照搬既有介面收斂點的設計意圖，簽名可依 GDScript
  慣例調整）。
- **規格來源**：`specs/SAVE_SCHEMA.md`
- **前置依賴**：CORE-1、CORE-2（`g_party` 成員要能呼叫 CORE-2 的 `derive()` 等價物，見 CORE-6/MOD-F）
- **驗收標準**：單元測試涵蓋背包加減/旗標讀寫的邊界情況（負數 clamp 到 0 等，照搬 `invAdd` 的
  `if(v[id]<0)v[id]=0` 規則）。
- **擁有檔案**：`godot-project/autoload/game_state.gd`。

---

## CORE-5　場景/轉場框架

- **目標**：取代 GDevelop 的 `replaceScene` + `g_result`/`g_returnScene`/`g_returnX/Y`/`g_spawn`/
  `g_encounter` 機制。
- **產出**：`godot-project/autoload/scene_router.gd`，提供 `go_to(scene_path, spawn_id)`／
  `start_battle(encounter_id, return_info)`／`battle_result(result: String)` 等 API，內部用 Signal 或
  直接呼叫 `get_tree().change_scene_to_file()`。
- **規格來源**：`specs/SAVE_SCHEMA.md`「場景轉場暫態值」一節；DEV_開發指南.md 的 boss/遭遇觸發段落
  （`g_returnScene/g_returnX/g_returnY` 語意）
- **前置依賴**：CORE-1、CORE-4
- **驗收標準**：World↔Battle 來回轉場、戰敗/逃走回到觸發前座標，跟 GDevelop 版行為一致（`specs/
  DIALOGUE_SPEC.md` D-6 的 boss 重試規則）。
- **擁有檔案**：`godot-project/autoload/scene_router.gd`。

## CORE-6　輸入抽象層

- **目標**：統一鍵盤與觸控輸入，對應 GDevelop 版「觸控 = 合成鍵餵給既有鍵盤流程」的設計（DEV 指南 L53）。
- **產出**：`godot-project/autoload/input_bridge.gd`，提供 `is_action_hit(action)` 等邊緣觸發判斷（對應
  GDevelop `keyHit()`），虛擬搖桿輸出映射到 Godot InputMap 的方向 action。
- **規格來源**：DEV_開發指南.md L53（觸控段落）
- **前置依賴**：CORE-1
- **驗收標準**：鍵盤與觸控（用 Godot 的 touch 事件模擬）都能觸發同一組 action，選單/商店/配點流程不用為
  觸控另寫一份判斷。
- **擁有檔案**：`godot-project/autoload/input_bridge.gd`。

---

## CORE-7　測試/驗證骨架

- **目標**：取代 `gdevelop-mcp`（`validate_project`/`preview_scene`）與 puppeteer E2E（`cq2_e2e.mjs`/
  `cq3_e2e.mjs`），建立 headless 可跑的驗證管道。
- **產出**：
  - `godot-project/tests/` 下的 GUT（Godot Unit Test）設定與至少一支 smoke test（載入 Title 場景無錯誤）
  - 一支 debug hook 腳本（對應 GDevelop 版 `window.__W`/`window.__B`/`window.__forceEnc`），暴露
    `GameState`/戰鬥狀態給自動化測試讀取
  - `TASKS/10_測試驗證.md` 詳列的 headless 自動打法腳本雛形
- **規格來源**：DEV_開發指南.md L73-79（測試章節）
- **前置依賴**：CORE-1~CORE-6（要有東西可測）
- **驗收標準**：`godot --headless -s tests/smoke_test.gd`（或 GUT 對應指令）可在 CI 環境跑過，不需要開視窗。
- **擁有檔案**：`godot-project/tests/**`。

## 執行順序摘要

```
CORE-1
  └─ CORE-2
       └─ CORE-4 ─┬─ CORE-3
                   ├─ CORE-5
                   └─ CORE-6
                        └─ CORE-7（等 MOD 任務有進度後再補測試也可，不硬性卡在最前面）
```

CORE-1/2 完成後，MOD-* 任務即可開始平行開工（各自的資料/場景骨架已就緒）；CORE-3/4/5/6 建議同一批完成後
再大量分派 MOD 任務，避免模組任務寫到一半發現 Autoload 介面還在變動。
