- Task 版本: v1.0
- 狀態: 草稿（尚未開工）
- 說明: 核心任務是所有模組任務（`01`~`09`）的地基，**必須依序完成**（CORE-1 → 2 → 3/4 → 5/6 → 7），

  - `godot-project/scenes/` `scripts/` `resources/content/` `assets/` `tests/` 目錄結構（已預先建好空目錄）

- **前置依賴**：無
- **驗收標準**：`godot --headless --check-only --path godot-project` 通過；編輯器內執行 title 場景無
  error/warning。
- **驗收現況（2026-07-13）**：本次執行環境沒有預裝 Godot，嘗試了以下管道下載 Godot 4.3+ linux 執行檔皆被
  網路 egress 政策擋下（非 TLS 問題，是 proxy 白名單機制，狀態碼多為 403 CONNECT rejected）：
  1. GitHub Releases 直接下載（`github.com/godotengine/godot/releases/download/...`）——被 proxy 攔截，
     回傳「GitHub access to this repository is not enabled for this session, use add_repo」；嘗試用
     `add_repo` 申請 `godotengine/godot` 存取被 auto-mode classifier 擋下（合理，`add_repo` 是給 clone
     原始碼用，且該工具本來就不會吐出 release 二進位資產）。
  2. 官方 tuxfamily.org 鏡像——`downloads.tuxfamily.org` 整個 host 被 proxy 政策擋下（`connect_rejected`）。
  3. `apt-cache search godot` 只有 `godot3`（3.5.2，Ubuntu noble universe），版本不符 CLAUDE.md 規定的
     4.3+（3.x 的 `.tscn`/GDScript/TileMap API 都不相容）。
  4. pip / npm 套件庫（`registry.npmjs.org`/`pypi.org` 這兩個 host 有明確放行）裡沒有真的引擎執行檔（查到的
     `godot`/`Godot` package 都是同名但無關的其他專案）。
  5. Docker Hub / GHCR 容器登錄檔的 manifest/token API（`registry-1.docker.io`、`auth.docker.io`、
     `ghcr.io`）本身可連得到，但實際 blob 內容存在 `production.cloudfront.docker.com` /
     `pkg-containers.githubusercontent.com`，這兩個 CDN host 一樣被政策擋下，所以 manifest 抓得到、
     實際二進位層下載不了。
  6. snapcraft / flathub / sourceforge 頂層網域同樣被擋。
  結論：這不是這個 agent 的操作失誤或 TLS 設定問題，是這次 session 的網路白名單本來就不包含任何能拿到 Godot
  官方二進位的路徑。**因此本任務目前用人工檢查取代二進位驗證**：
  - `project.godot`：手動核對 `[application]/[display]/[input]/[rendering]` 各 key 是否為 Godot 4.3
    官方 ProjectSettings 路徑與型別（例如 `window/stretch/mode`、
    `rendering/2d/snap/snap_2d_transforms_to_pixel`、`rendering/textures/canvas_textures/default_texture_filter`），
    並用小腳本核對 `{}`/`[]`/`()` 是否成對、9 個 InputMap action 區塊是否都正確收尾。
  - `scenes/title/title.tscn`：`[gd_scene format=3]` + 單一 `Node2D` 根節點，無腳本、無外部資源引用，
    是最保守、不可能出錯的合法 Godot 4 場景格式。
  - **這只是書面/語法層級的檢查，不等於實機驗證**；`window/stretch/mode="viewport"` 的選擇（見
    `CLAUDE.md`「渲染」一節）也還沒有實際開專案確認畫面縮放行為是否符合預期。
  - 下一步：任何有權限取得 Godot 4.3+ 執行檔的人（例如 John 本機或另一個沒有這個 egress 限制的環境）跑一次
    `godot --headless --check-only --path godot-project`，若通過即可把本任務狀態改為「已驗收」並勾選
    `MIGRATION_OVERVIEW.md`。
- **擁有檔案**：`godot-project/project.godot`、`godot-project/.gitignore`、頂層目錄結構本身。

## CORE-2　CONTENT.json → Godot Resource 轉存

- **狀態**：實作中（邏輯已完成並用純 Python 靜態比對驗證，但跟 CORE-1 一樣缺 Godot 執行檔做實機驗證——見下方
  「驗收現況」，尚未達到可打勾「已驗收」的門檻，需要有可用 Godot 4.3+ 執行檔的人/環境補跑一次
  `godot --headless --check-only` 並在編輯器內叫用 `ContentDB.get_enemy()` 等 API 確認）。
- **目標**：把 `../gd-crystal-tales/projects/crystal-quest/CONTENT.json`（party/derived/equipment/skills/
  pacing/items/enemies/encounters/shops/chests，共 8 個頂層分類、561 行；注意本文件先前寫「8 個頂層分類」但
  枚舉少了 `derived`/`pacing`，此處已依實際檔案內容修正）轉成 Godot 端可用的資料類別與讀取路徑，作為
  **唯一資料源**，之後改數值只改 CONTENT.json，不手動同步 Godot 端資料。
- **產出**：
  - `godot-project/scripts/content/` 下的 Resource 類別定義（`class_name` + `extends Resource`，皆含
    `from_dict()` 靜態工廠方法）：`PartyMemberDef`, `EquipmentDef`, `SkillDef`, `ItemDef`, `EnemyDef`,
    `EncounterDef`, `ShopDef`, `ChestDef`, `DerivedParams`, `PacingParams`。
  - 轉存腳本 `godot-project/scripts/content/sync_content.py`（Python，跟 `gd-crystal-tales` 那邊
    `art_v*.py` 系列一致風格）：讀 `CONTENT.json` → 輕量 schema 檢查（頂層分類/必要欄位是否齊全）→ 輸出
    `godot-project/resources/content/content.json`（帶 `_meta` 溯源欄位，記錄同步時間/來源路徑）。
  - `ContentDB`（autoload，`godot-project/autoload/content_db.gd`）：`_ready()` 時讀
    `res://resources/content/content.json` 並 `JSON.parse_string()` 建成上述 Resource 物件，提供
    `get_party_member(id)`／`get_equipment(id)`／`get_skill(id)`／`get_item(id)`／`get_enemy(id)`／
    `get_encounter(map_id)`／`get_shop(id)`／`get_chest(id)`／`get_derived()`／`get_pacing()` 及對應
    `get_all_*()` 查詢介面，模組任務一律透過這層介面拿資料，不直接讀檔案。
  - 驗收腳本 `godot-project/scripts/content/validate_content.py`（Python，不依賴 Godot 執行檔）：
    (1) 核對同步輸出跟來源 CONTENT.json 資料相等；(2) 靜態掃描每個 `*_def.gd` 的 `d.get("xxx", …)` 字面
    呼叫，回頭比對來源實際欄位，抓「讀了但來源沒有」的疑似打字錯誤；(3) 逐筆核對 id 唯一性與 enemies 的
    hp/atk/def/spd/exp/gold 等關鍵數值並列印供人工複核。
- **build-time 轉存 `.tres` vs run-time 直接 parse JSON —— 已定案：採用 run-time 直接 parse JSON**
  （CORE-2 執行時定案，回填本節，取代原本「取決於 CORE-2 實作時的取捨」的待定敘述）。理由：
  1. `CONTENT.json` 目前仍隨 GDevelop 端開發頻繁變動（John 常改數值），run-time parse 省去「改完
     CONTENT.json 還要在 Godot 編輯器內重新匯出/儲存 `.tres`」這道額外同步步驟，跟 `CLAUDE.md`
     「權威來源與資料流向」一節「不要手動同步改」的精神一致。
  2. `.tres` 需要 Godot 編輯器/執行檔才能產生與驗證；這次 CORE-2 執行環境（跟 CORE-1 一樣）拿不到 Godot
     執行檔，run-time parse 讓「純 Python 腳本驗證資料正確性」變得可行，不用等引擎起得來才能驗收。
  3. 目前資料量級小（561 行來源 JSON、最大分類 21 筆），run-time parse 一次的效能成本可忽略；
     `ContentDB` 的公開查詢介面（`get_xxx()`）刻意設計成跟資料來源方式無關，之後如果資料量大到需要優化，
     可以再補一支 build-time 轉存腳本，不影響呼叫端。
  - 代價：拿不到 `.tres` 的編輯器 Inspector 預覽與 Godot 原生型別檢查——這點在目前專案階段（資料驅動、
    無美術/關卡設計師需要在編輯器內直接編輯數值）不是主要工作流程，可接受。
- **規格來源**：`../gd-crystal-tales/projects/crystal-quest/CONTENT.json`（現有 schema，561 行，8 個頂層
  分類：`party`/`derived`/`equipment`/`skills`/`pacing`/`items`/`enemies`/`encounters`/`shops`/`chests`
  ——注意這樣數下來其實是 10 個分類，本文件先前寫「8 個」是沿用舊估算，已在此更正但不動版本號枝節文字，
  純粹補實際筆數）。
- **前置依賴**：CORE-1
- **驗收標準**：`ContentDB` 載入後，`get_enemy(id)` 等既有 id 能拿到跟 CONTENT.json 原始資料一致的值
  （原文舉例 `get_enemy("goblin_boss")`，但來源資料裡實際 id 是 `goblin_chief`，此處以實際資料為準）；
  寫一支對照腳本核對數量/關鍵欄位。
- **驗收現況（2026-07-13）**：
  - `godot-project/scripts/content/sync_content.py`：實跑通過，schema 檢查全過，輸出
    `godot-project/resources/content/content.json`，分類筆數
    `party=3 derived=18 equipment=21 skills=5 pacing=2 items=18 enemies=15 encounters=8(map) shops=2
    chests=12`。
  - `godot-project/scripts/content/validate_content.py`：實跑通過，`FAIL=0 WARN=0`。三層檢查（同步資料
    相等性／`.gd` 欄位讀取跟來源 schema 的靜態交叉比對／逐筆 id 唯一性＋enemies 六個關鍵數值列印）皆通過，
    細節見腳本輸出與檔頭註解。
  - **這只證明「轉存腳本輸出正確」＋「`.gd` 檔案裡的欄位字面字串跟來源 JSON key 對得上」，不等於實機驗證**：
    沒有 Godot 執行檔可以真的把 `content_db.gd` 跑起來，確認 `JSON.parse_string()` 解析結果、
    `Dictionary.get()` 型別轉換（`int()`/`float()`/`bool()`）、`class_name` 全域註冊在 Godot 4.3
    引擎裡實際行為都跟書面預期一致（例如 GDScript 對 JSON 數字解析成 int 還是 float 的邊界情況、
    `Resource.new()` 在 autoload `_ready()` 內建立大量物件有沒有非預期行為）。
  - 下一步：跟 CORE-1 一樣，任何有權限取得 Godot 4.3+ 執行檔的人跑一次
    `godot --headless --check-only --path godot-project`，並在編輯器/headless script 內呼叫
    `ContentDB.get_enemy("goblin_chief").hp` 等既有 id 確認回傳值正確，即可把本任務狀態改為「已驗收」。
- **擁有檔案**：`godot-project/scripts/content/**`、`godot-project/autoload/content_db.gd`、
  `godot-project/resources/content/**`。另因 CORE-1 的 `project.godot` 尚未有 `[autoload]` 區塊，
  CORE-2 在其中新增了該區塊並註冊 `ContentDB`（僅新增區塊，未動 CORE-1 既有的
  `[application]`/`[display]`/`[input]`/`[rendering]` 內容；理由與 CLAUDE.md 對應章節見
  commit message）。

## CORE-6　輸入抽象層

- **狀態**：實作中（`input_bridge.gd` 已完成並人工核對語法，但無法用 `godot --headless --check-only`
  實機驗證——跟 CORE-1/2/4/5 同一種環境限制，見下方「驗收現況」，尚未達到可打勾「已驗收」的門檻）。
- **目標**：統一鍵盤與觸控輸入，對應 GDevelop 版「觸控 = 合成鍵餵給既有鍵盤流程」的設計（DEV 指南 L53）。
- **產出**：`godot-project/autoload/input_bridge.gd`，提供 `is_action_hit(action)` 等邊緣觸發判斷（對應
  GDevelop `keyHit()`），虛擬搖桿輸出映射到 Godot InputMap 的方向 action。
- **規格來源**：DEV_開發指南.md L53（觸控段落）
- **前置依賴**：CORE-1
- **驗收標準**：鍵盤與觸控（用 Godot 的 touch 事件模擬）都能觸發同一組 action，選單/商店/配點流程不用為
  觸控另寫一份判斷。
- **驗收現況（2026-07-14）**：
  - **API 設計**：`InputBridge`（autoload，已在 `project.godot` [autoload] 區塊登記一行）提供五個
    方法：`is_action_hit(action)`（對應 GDevelop `keyHit()`，薄轉發 `Input.is_action_just_pressed()`）、
    `is_action_held(action)`（對應 GDevelop `hit`，薄轉發 `Input.is_action_pressed()`）、
    `is_action_released_this_frame(action)`（對稱介面，薄轉發 `Input.is_action_just_released()`，
    目前無強制呼叫端）、`simulate_action_press(action, strength=1.0)`／`simulate_action_release(action)`
    （對應 GDevelop `b.simulateControl()`，給 MOD-D 之後的虛擬搖桿/按鈕 UI 節點呼叫）。
  - **刻意不重做邊緣觸發追蹤**：`Input.is_action_just_pressed()` 語意已經跟 `keyHit()`（`b.kp[k]` 上一幀
    狀態比對）完全一致，`is_action_hit()`/`is_action_held()` 都是薄轉發，沒有自訂狀態表，因此本任務判定
    「邏輯複雜度低」，未另外寫 Python 交叉驗證腳本（依協調者指示：只有非平凡自訂邏輯才需要）。
  - **觸控/虛擬搖桿設計**：不維護獨立的觸控鍵狀態表（GDevelop 的 `st.tk` 集合在 Godot 端沒有對應實作
    必要）。改讓觸控 UI 節點直接呼叫 `simulate_action_press()`/`simulate_action_release()` 把觸控輸入
    寫回**同一組** InputMap action 的引擎狀態，`is_action_hit()`/`is_action_held()` 自然對觸控輸入生效。
    內部選用 `Input.action_press()`/`Input.action_release()`（而非建構 `InputEventAction` 丟
    `Input.parse_input_event()`）：前者是 Godot 官方文件明列給「非硬體來源模擬輸入」用的 API，立即同步
    更新 `Input` 單例狀態，不需要等事件佇列下一輪派發，時序上更貼近 GDevelop `st.tk`「當幀寫入、當幀
    可讀」的行為；詳細理由見 `input_bridge.gd` 檔頭註解。
  - **未新增/未修改 InputMap**：確認 `project.godot` `[input]` 區塊既有 action（`move_up/down/left/
    right`、`menu_toggle`、`stat_str/agi/int`、`battle_auto`）加上 Godot 內建 `ui_accept`/`ui_cancel`
    已覆蓋 DEV_開發指南.md L53 提到的所有觸控 UI 物件對應動作（JoyBase/Knob→方向、BtnA→ui_accept、
    BtnMenu→menu_toggle、PadU/D/L/R→方向、BtnBack→ui_cancel、BtnS1/2/3→stat_str/agi/int），**沒有發現
    遺漏的 action**，因此未修改 `[input]` 區塊；`[autoload]` 區塊只新增 `InputBridge="*res://autoload/
    input_bridge.gd"` 一行登記（比照 CORE-2/MOD-A 先例），未動其他既有登記內容。
  - **已核對 MOD-B 檔案無輸入依賴**：`scripts/world/trigger_zone.gd`／`exit_zone.gd`／`pickup_zone.gd`／
