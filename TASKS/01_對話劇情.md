# MOD-A　對話／劇情系統

- Task 版本: v1.0
- 狀態: 草稿
- 對應 GDevelop 系統: DEV_開發指南.md L48「對話/劇情佇列」
- 規格來源: `specs/DIALOGUE_SPEC.md` D-1、D-2、D-3

## 目標

移植 NPC 對話（`DLG`）與過場劇情（`CUTS`）系統：旗標比對選台詞、對話框逐句播放、`action` 副作用執行。

## 產出

- `godot-project/scripts/dialogue/dialogue_system.gd`：`open_npc_dialogue(npc_id)`、`play_cutscene(cut_id)`、
  逐句推進（對應 `st.dlgIdx`/`st.cutIdx`）。
- `godot-project/scripts/dialogue/flag_matcher.gd`：`specs/DIALOGUE_SPEC.md` D-1 的純函式實作（MOD-B 共用，
  誰先開工誰先建立，另一方直接引用，不要各寫一份）。
- `godot-project/resources/content/dialogue.json`（或 `.tres`）：從 `build_cq2.py` L931-990+ 抄出的 DLG/CUTS
  資料，附抽取腳本或手動轉寫紀錄。
- 每個 `action` id 的副作用實作（`register`/`ch1_take`/`ch1_reward`/`ch2_take`/`ch2_report`/`heal`/
  `give_ring`/`shop_hank_gift`/`shop_gid_gift`/`mira_start`/`mira_reward`/`relic_turnin`/`shop_hank`/
  `shop_gid` — 完整清單以實際回讀原始碼為準，不要只憑本檔案列的清單，可能有遺漏）。
- 對話框 UI 場景 `godot-project/scenes/ui/dialogue_box.tscn`（畫面呈現部分若 MOD-D 已建立通用文字框元件，
  這裡直接複用，不要重複做一個)。

## 前置依賴

CORE-1、CORE-2、CORE-4（讀寫 `GameState.flags`/`GameState.itemInv`/`GameState.gold`）、CORE-5（部分 action
會觸發存檔或轉場，如 `register` 之後可能有 UI 提示但不轉場；純粹保險起見依賴 CORE-5 API 存在）。

## 驗收標準

- 抽出的 `dialogue.json` 資料筆數與 `build_cq2.py` 現有 `DLG`/`CUTS` 條目數一致（人工核對一次）。
- 每個 NPC 在不同旗標狀態下對話內容與 GDevelop 版一致（抽樣至少涵蓋緹娜/漢克/老葛雷三位，因為他們的分支
  最多）。
- 每個 `action` 的旗標/道具/金幣異動跟原始碼一致。

## 擁有檔案

`godot-project/scripts/dialogue/**`、`godot-project/resources/content/dialogue.json`。
`flag_matcher.gd` 若 MOD-B 先建立則由 MOD-B 擁有，MOD-A 只能呼叫不能改（見 `11_並行協作規則.md`）。

## 已知風險

- DLG/CUTS 目前混在 3487 行的 `build_cq2.py` 裡，抽取時容易漏抄或抄錯行，建議寫一支一次性 Python 腳本
  直接 parse `build_cq2.py` 的 `DLG =`/`CUTS =` dict literal 並輸出 JSON，而不是手動複製貼上（人工抄容易
  出錯，且 GDevelop 端之後若還有台詞異動，重跑腳本就能重新抽取）。
