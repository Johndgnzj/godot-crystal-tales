# MOD-C　角色移動與碰撞

- Task 版本: v1.1
- 狀態: 實作中
- 對應 GDevelop 系統: `TopDownMovementBehavior::TopDownMovementBehavior`（game.json 內 5 個場景各用一次於
  Player 物件）、DEV_開發指南.md 的「隊伍跟隨（trail）」段落

## 目標

用 `CharacterBody2D` 重建頂視角八方向移動＋隊伍跟隨（trail）＋地圖碰撞（`BLK` 字串定義的阻擋格）。

## 產出

- `godot-project/scripts/world/player_controller.gd`：讀 CORE-6 的 `InputBridge` 輸出，
  `CharacterBody2D.move_and_slide()`，速度值需與 GDevelop `TopDownMovementBehavior` 的既有手感參數對齊
  （實作時在編輯器內用 GDevelop 版截圖/錄影比對移動速度手感，非公式化規格，記錄比對結論於本檔案）。
- `godot-project/scripts/world/party_trail.gd`：隊伍跟隨邏輯（記錄玩家歷史位置點陣列，後排角色依序沿路徑
  移動，對應 DEV 指南「trail」）。
- 碰撞：`BLK` 字串（build_cq2.py 地圖生成的阻擋格資料）轉成 Godot `TileMapLayer` 的碰撞層（實際轉換屬於
  MOD-H 範疇，本模組只消費 MOD-H 產出的碰撞層，不重新產生地圖資料）。

## 前置依賴

CORE-1、CORE-6；地圖碰撞資料依賴 MOD-H 的產出（可先用假資料/單一測試場景開發，之後接上 MOD-H 真正輸出）。

## lock/inside/cur_door 狀態設計（本次任務定案）

`autoload/game_state.gd`（CORE-4）檔頭已經查證原始碼並判斷這三個欄位屬於「世界場景控制器」職責、不進
GameState，交給 MOD-C 認領時決定放哪。定案：**per-scene 物件（`WorldSceneState`），不是 autoload**。

理由：
1. 貼近 GDevelop `rs.__v` 本來就是 per-scene runtime state 的語意（`if(!rs.__v){rs.__v={...}}`，見
   build_cq2.py:1389）——每個世界場景各自一份，換場景不需要手動歸零，天生避免「忘記重置導致舊場景狀態
   污染新場景」這類 bug；做成 autoload 反而要重新引入這個風險。
2. 沒有任何其他 MOD 任務需要跨場景讀取這三個欄位（衝突矩陣裡也沒有依賴方），不需要 autoload 的全域可見
   特性。
3. 用 `RefCounted`（不是 `Node`）：不需要進場景樹、不需要 `_process`，世界場景根節點（MOD-H 之後建立）
   在 `_ready()` 呼叫 `WorldSceneState.new()` 建立一份、指派給 `player_controller.world_state`、並用
   `register_gated_zone()`/`register_gated_zones()` 把場景內的 `ExitZone`/`PickupZone` 節點註冊進來即可。

`WorldSceneState` 提供的介面：
- `lock: bool` / `set_lock(bool)`：對應 GDevelop `lock`。
- `inside: bool`、`cur_door: Dictionary`（含 `tx`/`ty`）／`enter_building(door)`、
  `exit_building() -> Dictionary`：對應 `st.inside`/`st.curDoor`/`enterBuilding()`/`exitBuilding()`。
- `get_save_position(fallback, tile_size=32) -> Vector2`：室內時回傳門口外座標，供世界場景根節點呼叫
  `SaveManager.save_game()` 前校正座標（對應 build_cq2.py:1281；`SaveManager` 本身刻意不做這個判斷，
  見 `autoload/save_manager.gd` 檔頭「介面設計」一節）。
- `register_gated_zone(zone)` / `register_gated_zones(zones)`：註冊 `ExitZone`/`PickupZone` 節點，
  `lock`/`inside` 變動時自動同步它們的 `enabled` export（對應 GDevelop `!lock && !st.inside` 閘門，見
  build_cq2.py:2305）。

  gd` 目前只提供 `inside`/`cur_door` 狀態欄位本身，室內移動碰撞邏輯留給 MOD-H 資料就位後由後續任務（可能
  仍是 MOD-C 範圍，也可能併入 MOD-H）接上。
