# MOD-F　衍生屬性／戰鬥公式引擎

- Task 版本: v1.0
- 狀態: 草稿
- 對應 GDevelop 系統: WORLD_JS 與 BATTLE_JS 各一份的 `derive()`（DEV_開發指南.md L63：「改公式要同步兩處」
  ——這是本模組刻意要修掉的技術債）
- 規格來源: `specs/BATTLE_FORMULAS.md` F-1、F-2（全文）

## 目標

**唯一一份**衍生屬性計算（取代 GDevelop 版重複維護的兩份），供 MOD-C（世界場景 HUD 顯示屬性）與 MOD-E
（戰鬥數值）共用。

## 產出

- `godot-project/scripts/content/derive.gd`：純函式 `derive(member: PartyMemberData) -> PartyMemberData`，
  完整實作 `specs/BATTLE_FORMULAS.md` F-1（maxhp/maxmp/patk/matk/pdef/mdef/dodgeV/critV/spd 全部欄位，
  含裝備加成 `eqStat`、`critV` 取一位小數的刻意統一寫法）。
- `godot-project/scripts/content/exp_need.gd`：F-2 公式。
- 單元測試：至少覆蓋「無裝備」「滿裝備」「等級邊界」三種情境，數值跟 GDevelop 版現有角色實測比對（在
  GDevelop 版用 debug hook `window.__W`/`__B` 讀出實際數值當作測試 fixture，寫進測試檔案）。

## 前置依賴

CORE-2（需要 `ContentDB` 提供 `CONTENT.derived` 係數表與裝備資料）。

## 驗收標準

- 抽樣至少 3 名隊伍成員（不同 `mainAttr`/職業）在多個等級點的衍生數值，跟 GDevelop 版 debug hook 讀出的
  實測值一致（誤差為 0，因為是確定性公式，不接受「差不多」）。
- MOD-E、MOD-C 兩邊都改成呼叫這一份，不允許任何模組自己重算衍生屬性。

## 擁有檔案

`godot-project/scripts/content/derive.gd`、`exp_need.gd`。

## 已知風險

- 這是**唯一一個相對於 GDevelop 現況做刻意行為修正**（合併兩份 `derive()`、統一 `critV` 取一位小數）的
  模組，實作前務必跟 John 確認這個修正沒問題（理論上是純技術債清理、不影響玩家可見數值，但穩妥起見在
  `MIGRATION_OVERVIEW.md` 進度表勾選前先過一次目視確認）。
