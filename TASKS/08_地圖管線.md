# MOD-H　地圖生成管線

- Task 版本: v1.0
- 狀態: 草稿
- 對應 GDevelop 系統: `build_cq2.py` 第 4 節（MapB/carve_maze/make_forest，DEV_開發指南.md L87）＋
  tmj 輸出（`tmj_write`，L926-928）

## 目標

把地圖生成邏輯（迷宮雕刻、城鎮建築佈局 `BLDG_LAYOUT`、森林生成 `make_forest`）的**演算法**保留，輸出目標
從 GDevelop tmj/atlas 換成 Godot `TileMapLayer` + `TileSet` 資源，並保留「build 過 = 路通」的連通性 assert。

## 產出

- 一支 Python 或 GDScript `@tool` 腳本（建議 Python，跟現有 build 腳本風格一致，方便沿用既有迷宮生成
  程式碼骨架）：讀取跟 GDevelop 版相同的佈局規則（`BLDG_LAYOUT` 等常數需要抄過來或改成共用資料檔）→
  輸出 Godot 場景檔（`.tscn`，內嵌 `TileMapLayer` 節點）或 `.tres` 地圖資料。
- 六個場景各自的地圖輸出：Town/Forest/Forest2/Mine/Cave（Title/Battle 不需要地圖）。
- **連通性 assert 原樣保留**：build 產出後跑一次可達性檢查（BFS/DFS 從出生點能到所有關鍵點），過了才視為
  合法輸出，DEV 指南這條規則「改地圖後 build 過＝路通」對 Godot 版一樣適用。
- `BLK`（阻擋格）資料轉成 `TileSet` 的碰撞層設定，供 MOD-C 消費。
- `ENC`（高草/碎石遭遇地形）資料轉成自訂 metadata 或獨立 TileMapLayer，供 MOD-G 消費。

## 前置依賴

CORE-1；不依賴其他 CORE，可以較早開工（資料生成不需要 GameState/SaveManager 就緒）。

## 驗收標準

- 六張地圖生成後連通性 assert 全過。
- 圖塊視覺與 GDevelop 版 atlas 對應（不要求像素級全同，但地形/建築配置需一致，可用 GDevelop 版截圖比對）。
- `BLK`/`ENC` 資料正確標記，MOD-C/MOD-G 能讀到。

## 擁有檔案

新增地圖生成腳本（路徑待 CORE-1 目錄結構定案後補）、`godot-project/scenes/world/**` 下的地圖場景檔、
`godot-project/resources/**` 下的 TileSet 資源。**不要**修改 `../gd-crystal-tales/projects/crystal-quest/
scripts/build_cq2.py`（唯讀參考來源，見 CLAUDE.md 總則第 5 條）。

## 已知風險

- Godot 4.3 的 `TileMapLayer` API 跟 GDevelop 的 tmj/atlas 概念不是一對一對應，直接轉檔可能行不通，
  **建議走「重新生成」而非「轉檔」路線**：保留生成演算法（迷宮雕刻/建築擺放邏輯），輸出端從零開始寫成
  Godot 原生格式，理由已寫在 `MIGRATION_OVERVIEW.md`「需要重寫」表格。
- 裝飾 prop 的 `foot=(0,0,0,0)` 坑（會被當成整格是牆）是 GDevelop 版的既有陷阱，Godot 版重新設計 API 時
  可以順便避免這個坑（例如碰撞矩形留空就代表「無碰撞」而不是「零面積的牆」），但這屬於 API 設計改善，記得
  在文件裡註明新舊行為差異，避免其他任務誤以為兩邊語意相同。
