## 目標

把地圖生成邏輯（迷宮雕刻、城鎮建築佈局 `BLDG_LAYOUT`、森林生成 `make_forest`）的**演算法**保留，輸出目標
從 GDevelop tmj/atlas 換成 Godot `TileMapLayer` + `TileSet` 資源，並保留「build 過 = 路通」的連通性 assert。

## 產出

- 一支 Python 或 GDScript `@tool` 腳本（建議 Python，跟現有 build 腳本風格一致，方便沿用既有迷宮生成
  程式碼骨架）：讀取跟 GDevelop 版相同的佈局規則（`BLDG_LAYOUT` 等常數需要抄過來或改成共用資料檔）→
  輸出 Godot 場景檔（`.tscn`，內嵌 `TileMapLayer` 節點）或 `.tres` 地圖資料。
- 六個場景各自的地圖輸出：Town/Forest/Forest2/Mine/Cave（Title/Battle 不需要地圖）。
- **連通性 assert 原樣保留**：build 產出後跑一次可達性檢查（BFS/DFS 從出生點能到所有關鍵點），過了才視為
  合法輸出，DEV 指南這條規則「改地圖後 build 過＝路通」對 Godot 版一樣適用。
- `BLK`（阻擋格）資料轉成 `TileSet` 的碰撞層設定，供 MOD-C 消費。
- `ENC`（高草/碎石遭遇地形）資料轉成自訂 metadata 或獨立 TileMapLayer，供 MOD-G 消費。

## 前置依賴

CORE-1；不依賴其他 CORE，可以較早開工（資料生成不需要 GameState/SaveManager 就緒）。

## 驗收標準

- 六張地圖生成後連通性 assert 全過。
- 圖塊視覺與 GDevelop 版 atlas 對應（不要求像素級全同，但地形/建築配置需一致，可用 GDevelop 版截圖比對）。
- `BLK`/`ENC` 資料正確標記，MOD-C/MOD-G 能讀到。

## 擁有檔案

新增地圖生成腳本（路徑待 CORE-1 目錄結構定案後補）、`godot-project/scenes/world/**` 下的地圖場景檔、
`godot-project/resources/**` 下的 TileSet 資源。**不要**修改 `../gd-crystal-tales/projects/crystal-quest/
scripts/build_cq2.py`（唯讀參考來源，見 CLAUDE.md 總則第 5 條）。

## 已知風險

- Godot 4.3 的 `TileMapLayer` API 跟 GDevelop 的 tmj/atlas 概念不是一對一對應，直接轉檔可能行不通，
  **建議走「重新生成」而非「轉檔」路線**：保留生成演算法（迷宮雕刻/建築擺放邏輯），輸出端從零開始寫成
  Godot 原生格式，理由已寫在 `MIGRATION_OVERVIEW.md`「需要重寫」表格。
- 裝飾 prop 的 `foot=(0,0,0,0)` 坑（會被當成整格是牆）是 GDevelop 版的既有陷阱，Godot 版重新設計 API 時
  可以順便避免這個坑（例如碰撞矩形留空就代表「無碰撞」而不是「零面積的牆」），但這屬於 API 設計改善，記得
  在文件裡註明新舊行為差異，避免其他任務誤以為兩邊語意相同。

## 實作紀錄（v1.1，2026-07-14）

### 產出檔案

- `godot-project/scripts/map/gen_maps.py`（生成腳本，重跑整批重生，輸出具確定性）
- `godot-project/scenes/world/{town,forest,forest2,mine,cave}.tscn`（五張室外地圖；路徑對齊
  `SceneRouter.SCENE_PATHS`。Title/Battle 不含地圖，原文「六張」實為五張＋Battle 免地圖）
- `godot-project/resources/map/tileset_world.tres` / `tileset_forest.tres`
- `godot-project/scripts/world/world_scene.gd`（五張地圖共用的場景控制器，per-scene 差異走 @export，
  由生成器填入 .tscn）

### 採用的生成路線（「已知風險」的裁量結果：重新生成，非轉檔）

演算法（`MapB`/`carve_maze`/`BLDG_LAYOUT`/`make_forest`）自 build_cq2.py 第 4 節逐段移植，
**global RNG 消耗順序與原腳本完全一致（seed=51）**，因此可與 GDevelop 版 tmj 逐格比對。輸出端：

1. **地面＝TileMapLayer ＋ TileSet .tres（僅 atlas source）**。tile 資料以 `PackedInt32Array`
   export 存在場景根節點，`world_scene.gd._ready()` 以 `set_cell()` 填入——不手工生成
   `tile_map_data` 二進位 blob（格式風險高、本環境無 Godot 執行檔可驗證）。
2. **碰撞＝BLK 阻擋格貪婪合併成 StaticBody2D 矩形，直接烘進 .tscn**（每張圖 66~168 個矩形）。
   BLK 是 per-cell 資料（樹/建築擋在草地上），無法映射到 per-tile-type 的 TileSet physics，
   所以驗收標準「BLK 轉成 TileSet 碰撞層」照實際情況改走此路線；MOD-C 的 CharacterBody2D
   直接吃 StaticBody2D，語意相同。
3. **ENC＝沿用 CFG.ENC 字串陣列**存場景根節點 export，`world_scene.gd` 以
   `encounter_tracker.gd` 檔頭接法 (b)（callable 逐字元查表）餵給 MOD-G。
4. **視覺資源全部執行期載入＋`ResourceLoader.exists()` 檢查**（.tscn 不 ext_resource 任何貼圖）：
   MOD-I 資產分支未合併前場景照常載入運作（碰撞/出口/觸發/遭遇全部有效，只是無圖），合併後
   免改自動顯示。`foot=(0,0,0,0)` 陷阱在移植層保留原語意（BLK 需逐格一致），Godot 輸出層無此坑
   （碰撞從 BLK 網格烘出，prop 不帶碰撞概念）。

### 驗證結果

- **tmj 逐格比對全過**：`gen_maps.py` 內建與 GDevelop 版 tmj 產物比對，五張圖
  Town(1260)/Forest(2816)/Forest2(2816)/Mine(2520)/Cave(1800) 全部 `tmj_diff=0`。
- **連通性 assert 原樣保留且全過**（含寶箱擋格後重驗）；「build 過＝路通」規則對 Godot 版適用。
- 生成具確定性：重跑輸出 byte-identical（md5 比對），repo 內產物即最新生成結果、無手改漂移。
- `world_scene.gd` 已對照原始碼核對：出生點交握遵守 `scene_router.gd` 檔頭約定（讀
  `result`/`spawn`＋`should_use_return_position()`，定位完清空兩欄位）；戰後劇情三分支
  （story+Cave→`demon_post`、win+Mine+ch2==2+!c_mine_after→`mine_after`、lose→healAll+敗戰旁白）
  與 build_cq2.py L1435-1439 逐條一致；MOD-B zones 走 `WorldSceneState.register_gated_zone()`、
  MOD-G 走 `EncounterTracker.is_on_encounter_terrain` 接線、player 節點在 "player" group。
