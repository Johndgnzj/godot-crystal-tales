# 測試／驗證（收尾整合任務）

- Task 版本: v1.0
- 狀態: 草稿
- 對應 GDevelop 系統: gdevelop-mcp（`validate_project`/`preview_scene`）＋ puppeteer E2E（`cq2_e2e.mjs`/
  `cq3_e2e.mjs`）、DEV_開發指南.md L73-79

## 目標

建立 headless 可跑、可在 CI 重複執行的驗證管道，取代 GDevelop 端「開瀏覽器 + puppeteer 按鍵模擬」的做法，
並把 GDevelop 現有的 E2E 案例（DEV 指南 L73-79、ROADMAP Track A 的完整驗證流程）轉成 Godot 版回歸測試。

## 產出

- CORE-7 骨架之上，逐一補齊各模組的整合測試：
  - 世界場景 smoke test：載入六個場景無 error/warning。
  - 對話/劇情回歸：抽樣跑過 `specs/DIALOGUE_SPEC.md` 涵蓋的關鍵分支（緹娜/漢克/老葛雷的多階段對話）。
  - 戰鬥自動打法腳本：對應 GDevelop 版「等 `__B.state==="menu"` → Enter → `state==="target"` → Enter」
    的等價流程，改用 Godot 版 debug hook（CORE-7 產出）驅動。
  - 存檔/讀檔回歸：對照 `specs/SAVE_SCHEMA.md`。
  - 戰鬥公式回歸：對照 `specs/BATTLE_FORMULAS.md` F-1~F-9 的單元測試（MOD-F/MOD-E 各自已經產出，這裡只
    負責整合進統一的測試執行入口）。
- 一份「GDevelop 版 E2E 案例 → Godot 版測試」對照表，確保現有 ROADMAP.md 記錄過的驗證案例（例如 Track A
  「單場 puppeteer 從接委託打到回報，所有旗標/獎勵正確」）都有對應的 Godot 測試涵蓋，不遺漏既有回歸範圍。

## 前置依賴

CORE-7；實質內容依賴各 MOD 任務逐步完成，本任務**是持續性收尾工作，不是一次性任務**——每個 MOD 任務驗收時
都要順手在這裡補一條對應測試，不要留到最後才一次補。

## 驗收標準

- `godot --headless` 系列指令可在無 GUI 環境下跑完整套測試並回報 pass/fail。
- 每個 MOD-* 任務的「驗收標準」章節列出的檢查項目，都能在這裡找到對應的自動化測試或至少有手動驗證紀錄。

## 擁有檔案

`godot-project/tests/**`。

## 已知風險

- GDevelop 版的 debug hook（`window.__W`/`window.__B`/`window.__forceEnc`）是瀏覽器環境專屬設計，Godot
  版要重新設計等價機制（CORE-7 負責），不能照搬瀏覽器 API。
- 本任務容易因為「先把功能做出來，測試之後補」而被長期擱置——`TASKS/11_並行協作規則.md` 建議把「該模組是否
  已有對應測試」列進每個 MOD 任務的驗收 checklist，用制度性方式避免這個任務被無限期延後。
