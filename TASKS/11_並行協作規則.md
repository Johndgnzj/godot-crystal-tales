# 多 Subagent 並行協作規則

- Task 版本: v1.0
- 狀態: 定案
- 說明: 沿用 `../gd-crystal-tales/projects/crystal-quest/ROADMAP_開發計畫.md` 已驗證有效的「Track 並行
  批次」模式（該文件開頭的衝突規則），套用到本次 Godot 遷移的 CORE/MOD 任務上。

## 總則

1. **核心任務（CORE-1~7）互相有序，同一時間只允許一個 subagent 動 CORE 任務**（見
   `00_核心任務.md`「執行順序摘要」）。CORE 任務完成前，模組任務只能做讀規格/寫資料類別草稿等不需要整合
   執行的準備工作。
2. **模組任務（MOD-A~I）之間預設可平行**，除非下方「衝突矩陣」標記有共用檔案。
3. 每個任務的「擁有檔案」清單（見各 `TASKS/*.md`）是**唯一可以直接修改該檔案的任務**；其他任務如果需要
   擴充功能，透過該檔案已提供的公開介面（函式/Signal），不直接改內容。若介面不夠用，開一個新任務或跟
   擁有者任務的負責 agent 協調後再改，不要私自繞過。
4. 任務完成時：更新對應 `TASKS/*.md` 的「狀態」欄位為「已驗收」，並勾選 `MIGRATION_OVERVIEW.md` 的進度表。
5. 發現 spec（`specs/*.md`）與 GDevelop 原始碼實際行為不一致時，以原始碼為準，回頭更新 spec 並在 spec
   檔頭遞增版本號，附上發現日期與差異說明；不要各自依照自己讀到的舊 spec 各做各的。

## 認領任務時的檢查清單

開工前，subagent 應該：

1. 讀 `CLAUDE.md` 與本檔案。
2. 讀自己要做的 `TASKS/XX_*.md`，確認前置依賴的 CORE/MOD 任務狀態（讀對應檔案的「狀態」欄位，或問協調者）。
3. 檢查上方衝突矩陣，確認自己要碰的檔案沒有被其他「進行中」任務同時佔用。
4. 確認自己要引用的 `specs/*.md` 版本號，跟其他已完成/進行中任務引用的版本號一致（若不一致，代表 spec
   中途被更新過，要先確認是否影響自己的任務範圍）。

## 常見衝突情境與處理

- **兩個 MOD 任務都需要 `flag_matcher.gd` 但都還沒開工**：先起跑的任務負責建立，並在自己的 `TASKS/*.md`
  補一行「本任務已建立 flag_matcher.gd」，另一方開工前確認該檔案是否已存在，存在就直接引用。
- **MOD 任務發現 CORE 提供的 Autoload 介面不夠用**：不要私自加欄位/改簽名，改成在自己的任務檔案「已知風險」
  章節記錄需求，等 CORE 任務負責者統一處理擴充（避免多個任務各自用不同方式擴充同一個 Autoload 導致衝突）。
- **多個模組任務同時想改動同一張地圖場景檔**（例如 MOD-C 要調碰撞、MOD-G 要標 ENC 地形）：地圖場景檔由
  MOD-H 擁有，兩者都應該透過提 issue／在 `08_地圖管線.md` 補需求的方式，讓 MOD-H 任務負責者統一改動，不要
  兩邊直接編輯同一個 `.tscn` 檔案造成合併衝突。
