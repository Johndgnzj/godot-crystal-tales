## 一、道具的 icon／rarity

v2.0 替 `EquipmentDef` 加了第三個欄位 `attr_type`，見第七節。`ItemDef`／`EquipmentDef` 目前的欄位狀態
（`rarity` 的道具/裝備語意**不一樣**，v2.1 起特別注意）：

| 欄位 | 型別 | 說明 |
|---|---|---|
| `icon` | `String` | `res://` 圖示貼圖路徑；**留空字串＝美術尚未產出**，UI 端應 fallback 用分類（`cat`／`slot`）預設圖示，不是報錯 |
| `rarity`（裝備） | `String`（`@export_enum`） | 純 UI 呈現用稀有度色階：`common / uncommon / rare / epic`，不影響任何數值 |
| `rarity`（道具） | `String` | **v2.1 起與 PR #1 合併，改為實際掛在戰鬥掉落機制上**：`common / uncommon / rare / key`，
  搭配新欄位 `base_drop_rate: float` 一起決定掉落率，見第二節「與 PR #1 合併協調」與 `specs/BATTLE_FORMULAS.md` F-10 |

欄位都有預設值（`icon=""`、`rarity="common"`、`base_drop_rate=0.0`），不填就是安全預設值，不會因為沒改就壞掉。
`from_dict()` 同步讀取這些 key，供之後若 GDevelop 端也生出對應欄位時能無痛接上。

**為什麼不加「屬性」（元素/職業限定等玩法欄位）**：查過 `specs/BATTLE_FORMULAS.md`（衍生屬性/戰鬥公式，抄錄自
`build_cq2.py`）與現有 party/skill 資料，目前**沒有元素相剋、也沒有武器職業限定**這類機制。憑空加這類欄位會是
「發明新規則」而非「設計呈現層資料」，違反 `CLAUDE.md`「以原始碼為準，不要自行改規則」的精神，故本次刻意不加；
若之後要加元素/職業限定，屬於新的戰鬥機制設計，應該先開一個獨立任務並更新 `BATTLE_FORMULAS.md`。

## 二、稀有度設計

**⚠️ 道具（`ItemDef`）的 `rarity` 已在 v2.1 改由 PR #1（怪物平衡調整）的設計為準，本節原本 v1.0 訂的
`common/uncommon/rare/unique` 四級與判斷理由已不適用，改用下面「與 PR #1 合併協調」段落的版本。裝備
（`EquipmentDef`）的 `rarity` 不受影響，維持本文件自己的設計（見下表＋第八節）。**

裝備稀有度依「章節 tier ＋ 售價 ＋ 是否為指標武器」分級，`tier` 欄位保留（決定商店解鎖章節），`rarity`
是額外一層供 UI 呈現用，兩者不強制 1:1：

| 稀有度 | 判斷準則（tier／售價）|
|---|---|
| common（預設）| tier1 入門裝備，售價 ≤50G |
| uncommon | tier1 但售價較高（51~120G）／效果較全面的中階裝備 |
| rare | tier2（第二章）標準裝備 |
| epic | tier2 的「指標裝備」（如全裝備最高售價的主力武器、某流派最高單件加成）|

> 各裝備的實際稀有度值見 `equipment/*.tres`／codex，本表只記**分級準則**、不列具體品項（v2.2 去重）。

## 五、給設計員：新增道具/武器時怎麼填

延續 `設計員指南.md`「新增一個實體」流程：

1. 複製一個同分類現有 `.tres`、改檔名與 `id`。
2. `rarity` 依上表判斷邏輯挑一個對應等級（找不到規則可比照的先留 `common`，不要空想新等級）；`icon`
   若還沒有素材就留空字串，不要填一個不存在的路徑（Godot 載入不到會報錯）；裝備另有 `attr_type`
   （`str`/`agi`/`int`，僅 weapon/armor/boots 適用），見第七節。**新增道具**還要填 `base_drop_rate`
   （該道具想不想從戰鬥掉落／掉落基準率多少，`0.0` 表示不從戰鬥掉，只走商店/寶箱/劇情），比照
   `specs/BATTLE_FORMULAS.md` F-10 現行分級（普通素材 0.20、優良素材 0.12、稀有素材 0.05）。
3. ⚠️ **比 `設計員指南.md` 原文多一步、而且是關鍵一步**：光改 `resources/content/equipment/<id>.tres`（或
   `items/<id>.tres`）**這個檔案本身，遊戲讀不到**。原因與正確做法見下方第六節「重要架構備忘」，簡單說就是
   `content_db.tres` 裡有自己一份內嵌副本，兩邊都要改到才算數。

## 六、重要架構備忘：`content_db.tres` 才是遊戲實際載入的檔案

`設計員指南.md` 目前的說法是「改 `resources/content/<分類>/<id>.tres` → Inspector 改值 → 存檔即生效」。
本次擴充裝備數量、要把 116 筆裝備的資料寫回遊戲時，發現這句話**目前不成立**：

- `autoload/content_db.gd` 只 `load()` 一個檔案：`resources/content/content_db.tres`。
- 這個檔案裡的 `equipment`／`items`／`party`／…全部陣列，元素是**內嵌的 `[sub_resource]`**（自己完整一份
  `id`/`stats`/`desc`/… 拷貝），**不是**連到 `resources/content/equipment/<id>.tres` 這些個別檔案的
  `[ext_resource]` 連結。也就是說，`resources/content/equipment/short_sword.tres` 這個檔案跟
  `content_db.tres` 裡那份 `short_sword` 資料，是**兩份互相獨立的拷貝**，改一邊不會自動同步到另一邊。
- 這不是本次任務造成的，是先前 CORE-2/建置期間就存在的既有狀態（這個環境從頭到尾沒有真的 Godot 執行檔可以
  實機驗證 `build_tres.gd` 產出的檔案格式，`content_db.tres` 很可能是人工比照格式寫出來的，用內嵌
  sub_resource 而非跨檔案連結）。**v1.0（icon/rarity 上一輪）當時只改了個別 `.tres` 檔，沒同步
  `content_db.tres`——換句話說，上一輪加的 icon/rarity 資料，在遊戲實際載入的檔案裡其實是空的，這次已經
  一併補上同步（連同這次新增的 116 筆裝備）。**
- **本次做法**：兩邊都寫——個別 `.tres`（給編輯器 Inspector／未來人工編輯用）＋ `content_db.tres` 的
  內嵌 sub_resource（給遊戲實際 `load()` 用），確保現在兩邊資料一致。
- **後果／建議**：往後任何人只改個別 `.tres` 檔，**如果沒有同時開 Godot 編輯器讓它自動重新聚合、或手動
  同步 `content_db.tres`，改動不會反映在遊戲裡**。這跟 `設計員指南.md` 承諾的「Inspector 改值即生效」有
  落差，建議之後有 Godot 編輯器可用時實際測試一次「改個別 .tres → 存檔 → 重開遊戲」是否真的沒生效，如果
  確認落差存在，應該回頭把 `content_db.tres` 改成用 `[ext_resource]` 連結個別檔案（`build_tres.gd`
  理論上這樣做才對），或者更新 `設計員指南.md` 說明正確流程；這件事已經超出本次「擴充道具/裝備」任務的
  範圍，這裡先誠實記錄，不擅自動手重構整個聚合檔案的架構。

## 八、v2.0 裝備擴充：力量/敏捷/法力型 8 階曲線 ＋ 40 種飾品

武器/防具/腳部三個部位，各自拆成「力量型（str）／敏捷型（agi）／法力型（int）」三條路線，每條路線 8 階
（1~4 階 tier1，第一章鐵匠鋪 `hank` 常態販售；5~8 階 tier2，第二章解鎖，`shop.gd` 既有的
`tier>=2 需要 ch2 旗標` 閘門邏輯原樣適用，沒有新增判斷邏輯）。稀有度沿曲線遞增：1~2 階 common、3~4 階
uncommon、5~6 階 rare、7~8 階 epic。**既有裝備原地保留 id，插入曲線對應的階數**——例如 `steel_sword`
（鋼劍）原本是「當時最強」（epic），現在是力量型武器第 5 階（曲線中段），稀有度改標 `rare`，把 epic
讓給新增的第 7、8 階（`dragonfang_sword` 龍牙劍／`sacred_greatsword` 聖銀大劍）；這不是砍鋼劍強度
（`patk=10` 數值完全沒動），純粹是稀有度標籤跟著曲線重新定位。

`attr_type` 只是**分類標籤**，不是穿裝限制——沒有像 `specs/` 規格那樣查到「哪個角色只能用哪型武器」的
既有規則（目前只有 `str`/`agi` 兩個 mainAttr 角色，没有 int 主屬性角色），所以三型裝備任何角色都能裝備，
`attr_type` 純粹給 UI 未來做篩選/標籤用，跟 `slot` 的穿裝位置限制是兩回事。

飾品（`acc`，角色有 `acc1`/`acc2` 兩格可裝）不分型，40 種都是全能/雜項屬性（會心、閃避、生命、法力、
金幣類單一或雙屬性小加成，highest 到多屬性大加成），稀有度分布 common12／uncommon12／rare10／epic6。
