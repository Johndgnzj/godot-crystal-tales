# 道具與武器：屬性／圖示／說明 設計文件

- 文件版本: v1.0
- 對應 schema 快照: `godot-project/scripts/content/item_def.gd`、`equipment_def.gd`（本次新增 `icon`／`rarity` 欄位後）
- 狀態: 定案（schema＋資料已套用；圖示美術資產待後續美術任務補齊，見下方「已知缺口」）
- 性質說明: 本文件**不是**從 GDevelop `build_cq2.py` 抄錄的既有規則（不同於 `specs/` 底下三份規格），
  是 Godot 版新增的**呈現層**設計（稀有度色階、圖示欄位），不影響任何既有戰鬥/存檔數值公式。

## 一、這次擴充了什麼

`ItemDef`（`scripts/content/item_def.gd`）與 `EquipmentDef`（`scripts/content/equipment_def.gd`）原本只有
「效果／說明文字」（`effect`／`desc`）與交易數值（`buy`／`sell`），沒有圖示、也沒有稀有度分級。新增兩個欄位：

| 欄位 | 型別 | 說明 |
|---|---|---|
| `icon` | `String` | `res://` 圖示貼圖路徑；**留空字串＝美術尚未產出**，UI 端應 fallback 用分類（`cat`／`slot`）預設圖示，不是報錯 |
| `rarity` | `String`（`@export_enum`） | 稀有度色階，供未來 UI 用色框/底色區分。道具：`common / uncommon / rare / unique`；裝備：`common / uncommon / rare / epic` |

兩個欄位都有預設值（`icon=""`、`rarity="common"`），**現有 39 筆資料（18 道具＋21 裝備）不填就自動是「無圖示、
普通稀有度」**，不會因為沒改就壞掉。`from_dict()` 也同步讀取這兩個 key（`d.get("icon","")`／
`d.get("rarity","common")`），供之後若 GDevelop 端也生出對應欄位時能無痛接上。

**為什麼不加「屬性」（元素/職業限定等玩法欄位）**：查過 `specs/BATTLE_FORMULAS.md`（衍生屬性/戰鬥公式，抄錄自
`build_cq2.py`）與現有 party/skill 資料，目前**沒有元素相剋、也沒有武器職業限定**這類機制。憑空加這類欄位會是
「發明新規則」而非「設計呈現層資料」，違反 `CLAUDE.md`「以原始碼為準，不要自行改規則」的精神，故本次刻意不加；
若之後要加元素/職業限定，屬於新的戰鬥機制設計，應該先開一個獨立任務並更新 `BATTLE_FORMULAS.md`。

## 二、稀有度設計（已套用到全部 39 筆資料）

道具稀有度依「取得難度／效果強度」判斷（金幣價格是強訊號，`crystal_shard` 額外因為 `effect` 文字本身寫明
「稀有」而定為 rare）：

| 稀有度 | 道具 | 判斷依據 |
|---|---|---|
| common（預設，未寫入欄位） | 藥水、魔力草、解毒草、鐵礦石、哥布林牙、史萊姆凝膠、柔軟羽毛、白骨碎片、狼皮、蟲甲 | 基礎消耗品／常見素材，售價低、掉落率高 |
| uncommon | 上級藥水、復活羽毛、帳篷、獸皮 | 效果影響力較大（復活/全隊回復）或素材來源較強的敵人 |
| rare | 水晶碎片 | 效果文字明寫「稀有；似乎與礦山異變有關」，且是主線伏筆道具 |
| unique | 女神的記號、公會登錄證、失蹤礦工的頭盔 | 劇情/委託專用道具（`cat=="key"`），不可販售、獨一份 |

裝備稀有度依「章節 tier ＋ 售價 ＋ 是否為指標武器」分級，`tier` 欄位保留（決定商店解鎖章節），`rarity`
是額外一層供 UI 呈現用，兩者不強制 1:1：

| 稀有度 | 裝備 | 判斷依據 |
|---|---|---|
| common（預設） | 短劍、獵刀、布質護腕、瑪瑙護符、皮革長靴、旅行衣、幸運銅幣、獵人護腕 | tier1 入門裝備，售價 ≤50G |
| uncommon | 疾風靴、活力戒指、鐵護腕、凝神耳環、皮革護胸、鐵劍 | tier1 但售價較高（51~120G）／效果較全面的中階裝備 |
| rare | 銀靴、精鋼護腕、守護墜飾、鎖子甲、法師袍 | tier2（第二章）標準裝備 |
| epic | 銳鋒匕、鋼劍 | tier2 裡的「指標裝備」——鋼劍是漢克鍛造的第二章主力武器（售價全裝備最高 260G），銳鋒匕是會心
  流派最高單件加成（crit+8），各自代表一條主流派（物理輸出／會心爆擊）的頂點 |

## 三、圖示（icon）欄位與現況缺口

**現況**：專案裡目前只有一張可用的道具/裝備類圖示素材——`assets/battle/Red_Potion.png`（16×16 像素，
先前一直未被任何程式碼引用），已指定給 `items/potion.tres` 的 `icon` 欄位。**其餘 38 筆資料 `icon` 欄位維持
空字串**——沒有對應素材，不硬塞不相關的既有圖片（例如拿 `crate.png` 充當「鐵礦石」圖示會誤導玩家）。

**沒有在這次順便生圖的原因**：
1. `.claude/skills/gen-art/SKILL.md` 明文排除「16-64px 像素素材」（道具圖示屬於這個尺寸級距），水彩風的
   Gemini 生圖管線畫不出對齊的像素網格，該 skill 的做法是走 LPC 圖層合成或商店素材包。
2. 本次工作環境沒有 GDevelop 姊妹專案（`../gd-crystal-tales/`）可以讀取既有素材庫，也沒有網路權限抓取新的
   像素圖示包（`assets/battle/lpc_src/` 目前只有敵人立繪，沒有道具圖示來源）。

**圖示命名慣例**（供之後補圖的人依循，跟 `設計員指南.md`「敵人戰鬥圖 / 道具圖 → `assets/battle/`」的既有
慣例一致）：`res://assets/battle/icon_<id>.png`，`<id>` 對應 `.tres` 檔名（例：`icon_hi_potion.png`、
`icon_iron_sword.png`）。16×16 或 32×32 像素、與現有 `foe_*`/`Red_Potion.png` 同解析度基準，nearest filter。

**待補清單（38 筆，依優先順序）**：
- 優先：商店常態販售品（消耗品全部、tier1 武器/防具）——玩家最常在 UI 看到。
- 次要：材料類（賣店用，出現頻率較低）。
- 最後：`key` 分類劇情道具（通常整場只看到一次，急迫性最低）。

補圖後只需要在對應 `.tres` 的 `icon` 欄位填路徑，Godot 編輯器會自動 import，不需要改 schema。

## 四、道具/裝備「說明」欄位的既有狀況（本次未變動，記錄供設計員參考）

查證現有 UI 程式碼（`scenes/ui/menu_root.gd`、`scenes/ui/shop.gd`）後確認：

- **道具**的 `effect` 欄位就是玩家會看到的說明文字（選單「效果：」列、商店清單下方說明列都直接顯示它）。
  18 筆道具的 `effect` 已經是「消耗品寫機制、素材/劇情道具寫風味文字」的可用狀態，本次未改文字內容。
- **裝備**的 `desc`（風味文字）**目前沒有任何 UI 會顯示**——選單與商店都是呼叫 `_eq_desc()`
  依 `stats{}` 自動組出「物攻+6」這類數值列表，`desc` 欄位是「寫好但沒人念」的狀態。這不是本次改動造成的
  （查證 `build_cq2.py` 原版 `eqDesc()` 邏輯本來就只列數值），只是記錄下來：如果之後想讓裝備風味文字派上
  用場（例如道具詳情頁加一行斜體風味文字），`desc` 資料已經齊全，是 UI 端（MOD-D 範圍）要不要接的問題，
  不需要設計員重寫文案。

## 五、給設計員：新增道具/武器時怎麼填

延續 `設計員指南.md`「新增一個實體」流程，多兩個欄位可填（都非必填，留空即安全預設值）：

1. 複製一個同分類現有 `.tres`、改檔名與 `id`。
2. 照 `設計員指南.md` 掛進 `content_db.tres`。
3. **新增**：`rarity` 依上表判斷邏輯挑一個對應等級（找不到規則可比照的先留 `common`，不要空想新等級）；
   `icon` 若還沒有素材就留空字串，不要填一個不存在的路徑（Godot 載入不到會報錯）。

## 六、已知風險：`content.json` 不同步

`resources/content/content.json` 是舊的匯入種子（`sync_content.py` 從 GDevelop `CONTENT.json` 產出），
`icon`/`rarity` **沒有**寫回這個檔案——`設計員指南.md` 明確要求「數值請改 `.tres`，不要去改
`resources/content/content.json`」，本次遵循此規則。代價：如果之後有人重跑 `build_tres.gd`
（從 `content.json` 全量重建 `.tres`，僅在「要從 GDevelop 重新拉新資料」時才會做），本次填入的
`icon`/`rarity` 值會被 `from_dict()` 的預設值（`""`／`"common"`）覆蓋掉，需要照本文件第二、三節重新套用一次。
`scripts/content/validate_content.py` 已同步更新（新增 `GODOT_NATIVE_EXTRA_KEYS` 白名單），不會誤報
`icon`/`rarity` 是「疑似打錯字的欄位」。
